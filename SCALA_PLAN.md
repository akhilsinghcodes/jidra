# JIDRA Scala Support ÔÇö Implementation Plan

## Why Scala

JIDRA supports Java (~85% resolution), TypeScript (~80%), and Python (~68.5%). Scala is the next language because:

1. **SemanticDB** ÔÇö the Scala compiler itself emits fully-resolved symbol references for every method call. No re-inference needed. Expected resolution: **~90%**, the highest of any JIDRA language.
2. **Strong static types** ÔÇö unlike Python, Scala's type system is explicit and compiler-checked, so phantom edges are near zero.
3. **sbt ecosystem** ÔÇö most Scala projects use sbt, which makes detection and compilation deterministic.

---

## Architecture

Scala follows the **TypeScript pattern**: an ephemeral Docker sidecar compiles the project and emits extraction data, which Python reads into a `Graph`.

```
Java:   repo Ôćĺ tree-sitter (in-process) Ôćĺ Spring Actuator (runtime validation)
TS:     repo Ôćĺ Docker (ts-morph sidecar emits JSONL stdout)
Python: repo Ôćĺ ast (in-process) Ôćĺ Pyright (static validation)
Scala:  repo Ôćĺ Docker (sbt compile + semanticdb) Ôćĺ Python reads .semanticdb proto files
```

**Key difference from TypeScript:** The sidecar does not emit JSONL to stdout. Instead it runs `sbt compile` (which writes `.semanticdb` binary files under `target/semanticdb/`), and the Python extractor reads those proto files directly from a mounted output volume. This avoids piping binary protobuf through stdout.

---

## SemanticDB Overview

SemanticDB is generated by the Scala compiler plugin `semanticdb-scalac`. Each `.semanticdb` file is a protobuf `TextDocuments` message containing:

- **`occurrences[]`** ÔÇö every symbol use in the source file, with range + resolved symbol + role (REFERENCE or DEFINITION)
- **`symbols[]`** ÔÇö metadata for each symbol: kind (METHOD, CLASS, FIELD, etc.), signature, display_name, overridden_symbols

A method call site is a `SymbolOccurrence` with `role=REFERENCE` whose resolved symbol maps to a `SymbolInformation` with `kind=METHOD`. The symbol string looks like: `com/example/UserService#findById().` ÔÇö package, class, method name, and trailing `.` for methods.

This gives us **compiler-resolved call edges for free** ÔÇö no multi-phase re-inference like Python or Java.

## Phase 1 ÔÇö Proto Bindings

**Generate Python bindings** from the official scalameta proto:

```bash
pip install grpcio-tools
python -m grpc_tools.protoc \
  -I jidra/scala_proto \
  --python_out=jidra/scala_proto \
  jidra/scala_proto/semanticdb.proto
```

Commit the generated `semanticdb_pb2.py` so `grpcio-tools` is not a runtime dependency.

**Key proto types used at runtime:**

```python
from jidra.scala_proto import semanticdb_pb2

docs = semanticdb_pb2.TextDocuments()
docs.ParseFromString(path.read_bytes())

for doc in docs.documents:
    for occ in doc.occurrences:
        # occ.range.start_line, occ.symbol, occ.role
        if occ.role == semanticdb_pb2.SymbolOccurrence.REFERENCE:
            ...  # call site
    for sym in doc.symbols:
        # sym.symbol, sym.kind, sym.display_name, sym.signature
        if sym.kind == semanticdb_pb2.SymbolInformation.METHOD:
            ...  # method definition
```

## Phase 2 ÔÇö Docker Sidecar

**`scala_sidecar/Dockerfile`:**
```dockerfile
FROM sbtscala/scala-sbt:eclipse-temurin-21_1.10.0_3.4.2
WORKDIR /app
COPY entrypoint.sh .
RUN chmod +x entrypoint.sh
ENTRYPOINT ["/app/entrypoint.sh"]
```

**`scala_sidecar/entrypoint.sh`:**
```bash
#!/bin/bash
set -e
REPO=$1
OUTPUT=$2  # mounted output volume path

cd "$REPO"

# Inject semanticdb plugin into project/plugins.sbt if not already present
PLUGINS_FILE="project/plugins.sbt"
mkdir -p project
if ! grep -q "semanticdb" "$PLUGINS_FILE" 2>/dev/null && ! grep -q "semanticdb" build.sbt 2>/dev/null; then
  echo 'addCompilerPlugin("org.scalameta" % "semanticdb-scalac" % "4.17.0" cross CrossVersion.full)' >> build.sbt
  echo 'scalacOptions ++= Seq("-Yrangepos", s"-P:semanticdb:sourceroot:${baseDirectory.value}")' >> build.sbt
fi

# Compile (generates .semanticdb files under target/semanticdb/)
sbt compile 2>&1 >&2

# Copy all .semanticdb files to the output volume, preserving relative paths
find . -name "*.semanticdb" -type f | while read f; do
  dest="$OUTPUT/$f"
  mkdir -p "$(dirname "$dest")"
  cp "$f" "$dest"
done
```

**Python invocation (in `scala_extractor.py`):**
```python
import tempfile
```

## Phase 3 ÔÇö `scala_extractor.py`

Two-pass extraction:

**Pass 1 ÔÇö Definitions:** Iterate all `.semanticdb` files, collect `SymbolOccurrence` with `role=DEFINITION`. For each, look up `SymbolInformation` and create `ClassEntry` / `MethodEntry` / `FieldEntry`. Build a `symbol Ôćĺ MethodEntry` lookup dict.

**Pass 2 ÔÇö Call sites:** Iterate all `.semanticdb` files again, collect `role=REFERENCE` occurrences whose resolved symbol maps to a known `MethodEntry`. Create `CallSite` + `ResolvedCallEdge` pairs.

**ID generation:** Reuse `class_id`, `method_id`, `callsite_id`, etc. from `models.py` ÔÇö same SHA1 stable IDs as Java/TypeScript/Python.

**Package extraction:** Parse `package` declaration from first 10 lines of source file (regex `^package\s+(.+)`).

**Scala stereotypes:**

| Pattern | Stereotype |
|---|---|
| Annotation `@Controller` / extends `Controller` (Play) | `controller` |
| Annotation `@Singleton` / extends `Actor` (Akka) | `service` |
| Annotation `@Route` | `endpoint` |
| Scala `trait` (kind=TRAIT) | `trait` |
| Scala `object` (kind=OBJECT) | `object` |
| Filename `*Controller.scala` | `controller` |
| Filename `*Service.scala` | `service` |
| Filename `*Repository.scala` / `*Repo.scala` | `repository` |

**Scala 3 compatibility:** Detect Scala 3 via `build.sbt` (look for `scalaVersion := "3."` prefix). For Scala 3, use `-Xsemanticdb` compiler flag instead of the plugin.

## Revised Phase 4 ÔÇö Language Detection

`detect_languages()` in `ts_filters.py` already exists. Add Scala check **before** Java (Scala projects often also have `build.gradle` in mixed setups):

```python
def detect_languages(root: Path) -> list[str]:
    langs = []
    # Scala before Java ÔÇö build.sbt is the definitive Scala signal
    if (root / "build.sbt").exists() or (root / "project" / "build.properties").exists():
        langs.append("scala")
    if (root / "package.json").exists() or ...:
        langs.append("typescript")
    if (root / "pom.xml").exists() or ...:
        langs.append("java")
    if (root / "pyproject.toml").exists() or ...:
        langs.append("python")
    return langs
```

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| `sbt compile` slow on large projects | Default timeout 600s; show progress message |
| Injecting plugin breaks `build.sbt` syntax | Inject into `project/plugins.sbt` instead; check for existing semanticdb config first |
| Scala 3 uses different semanticdb flag | Detect `scalaVersion` in `build.sbt`; use `-Xsemanticdb` for `3.x` |
| `sbtscala/scala-sbt` image is ~235MB | Acceptable ÔÇö same tradeoff as TypeScript sidecar |
| Proto schema changes across scalameta versions | Pin to v4.17.0; commit generated `_pb2.py` |
| First-run sbt downloads dependencies (slow) | Document expected wait time; sbt cache persists in Docker layer |
| Scala + Java in same repo (Play with Spring libs) | Both detected; graphs merged; language tags disambiguate |
